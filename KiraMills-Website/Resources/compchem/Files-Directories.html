Part I. Definitions


Whether you’re working in Windows, MacOS, or a Linux environment, it’s important to understand file management, directories/files, and paths. A computer file is an electronically stored collection of characters or data. These files might be called documents, graphics, programs, etc. If you were to print one, you’d have a paper (or hard copy) file. You could put this file in a folder and then place the folder in a file cabinet with other folders. Whatever organization you imagine for a real file cabinet, can also be achieved in an electronic file cabinet. A collection of electronic files can be stored together in a directory or folder. There’s no real difference between these terms - Windows likes to use the word “folder”, and Linux likes to use the word “directory”. You can store folders within folders if you wish. To describe a directory that’s inside another directory, you might see the word subdirectory.
There are several ways you could show your file/directory/folder organization schematically. In Figure 1 the main folder is blue, there are two green folders inside of the main folder, and then a purple folder inside each green folder. Each folder contains at least one file, but they could be empty if we chose them to be. The main folder is called the root directory. In the hierarchical structure, the root directory is said to be the highest level, or top, directory, not the lowest one. This isn’t as clear in Figure 1, so we can instead depict the same information in an upside down tree (Figure 2). In this schematic, the blue root directory is the highest, the green directories are one level below it, and the purple directories are two levels below. We can describe the location of other directories or files using the words “down” and “up”. In reference to the green directory, the purple directories and their files are “down one level” and the blue directory and its file are “up one level”. In both schematics, the letter U in the file name means “up” from the green directory, and “D” means “down”.
One of our major concerns when working with files and directories, is being able to provide a path to a specific file/directory. That is, writing the exact instructions for how to get to the object of interest. For example, if we wanted to write the full path to file 2D.1 in the schematic above, we would want to start with the highest level and work our way down, so it would be: Root/Directory2/Directory2D/file2D.1
If we instead wanted to give the instructions for how to to file 2D.1 if we were starting in Directory 1, or in other words give the relative path, we would need to follow the tree between the two. As a note, when we want to “move up” a directory when giving a path, we use “../” to denote this, so our relative path would be: ../Root/Directory2/Directory2D/file2D.1
If you’re using a Windows computer, you can see the path to any file in the file explorer window. For example, in Figure 3 below, the path to my directory called Chem3321 is displayed in the top bar (but with arrows between directories instead of slashes) as This PC > Documents > TA > Chem3321. If I wanted to see this path in the same way we wrote out the paths above, I can right click in the bar, and select “Copy address as text”, which gives the following. Notice, an important difference between Linux and Windows computers, is that in Linux, we give the path using forward slashes ( / ), but in Windows, we give the path using backwards slashes ( \ ). C:\Users\kirar\Documents\TA\Chem3321
